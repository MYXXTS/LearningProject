# 共享单车锁项目说明（当前实现版）

## 1. 文档目的

本文档用于说明当前 `bike` 项目的真实实现状态，覆盖以下内容：

- 硬件接线与实际分工
- 单片机整机固件的工作流程
- Flutter App 的职责范围
- LCD1602 显示屏全部显示内容及含义
- 提示音曲目映射
- 当前项目的调试与使用要点

本文档基于当前代码实现整理，不再沿用旧版方案中未落地或已调整的描述。

## 2. 当前项目结构

- `firmware/01_bluetooth_lcd_debug`：蓝牙收发与 LCD 独立调试工程
- `firmware/02_gps_time_lcd`：GPS 时间读取独立调试工程
- `firmware/03_servo_keypad_debug`：舵机独立调试工程
- `firmware/04_mp3_keypad_debug`：MP3 模块独立调试工程
- `firmware/05_bike_lock_controller`：当前整机固件
- `app/`：Flutter Android App
- `docs/`：项目文档
- `reference/`：外设说明书、示例代码与参考资料

## 3. 硬件分工

### 3.1 单片机与外设

| 引脚 | 硬件 | 说明 |
| ---- | ---- | ---- |
| `P0.0-P0.7` | LCD1602A 数据总线 | `DB0-DB7` |
| `P2.5` | LCD1602A | `RW` |
| `P2.6` | LCD1602A | `RS` |
| `P2.7` | LCD1602A | `EN` |
| `P2.0` | SG90 舵机 | PWM 输出 |
| `P2.1/P2.2` | GPS 模块 | 当前整机流程中保留接线，但不参与功能 |
| `P2.3` | MP3-TF-16P | 软件 UART `TXD`，单片机发，模块收 |
| `P3.0` | HC-06 | 单片机硬件 UART `RXD`，接 HC-06 `TXD` |
| `P3.1` | HC-06 | 单片机硬件 UART `TXD`，接 HC-06 `RXD` |

### 3.2 当前功能归属

- 单片机负责：
  - 蓝牙命令接收与响应
  - 舵机开锁、关锁
  - MP3 提示音播放
  - LCD 状态显示
- App 负责：
  - 二维码解析与 MAC 提取
  - 蓝牙连接管理
  - 手机定位获取
  - 围栏设置与围栏判断
  - 开锁/关锁前置决策
  - 时间记录、骑行记录与费用计算
- GPS 模块当前不参与整机流程：
  - 仍保持接线
  - 不做定位
  - 不做授时
  - 不参与任何状态判断

## 4. 整机固件行为

当前整机固件为 `firmware/05_bike_lock_controller`，核心逻辑如下：

1. 上电初始化 LCD、舵机、蓝牙串口、MP3 模块。
2. 舵机默认进入上锁位。
3. 蓝牙收到命令后，按协议执行：
   - `U`：开锁
   - `L`：关锁
   - `S`：查询状态
   - `P,n`：播放指定提示音
4. 单片机不接收围栏参数，也不做任何定位计算。
5. 开锁/关锁执行时，固件内部会短暂进入忙状态，此时新收到的 `U/L/P` 会返回 `BUSY`。

## 5. App 行为

当前 App 为 Android 版本，主要页面和职责如下：

- 首页：
  - 扫码或粘贴二维码文本
  - 连接/断开 HC-06
  - 刷新定位
  - 查询状态
  - 开锁、关锁
  - 播放测试提示音
- 围栏页：
  - 配置圆形围栏
  - 配置多边形围栏
  - 使用当前位置填充围栏坐标
- 记录页：
  - 查看历史骑行记录
- 调试页：
  - 临时关闭围栏
  - 设置模拟当前位置
  - 设置模拟开锁位置
  - 设置模拟关锁位置
  - 播放 `001-005`
  - 清除当前骑行
  - 查看错误面板

## 6. LCD1602 显示说明

LCD 始终显示两行。

- 第 1 行：当前整机状态
- 第 2 行：最近一次动作或最近一次协议处理结果

### 6.1 启动画面

| 第 1 行 | 第 2 行 | 含义 |
| ---- | ---- | ---- |
| `BIKE LOCK INIT` | `BT+SERVO+MP3` | 固件启动完成，正在展示整机初始化画面 |

该画面显示约 1 秒，然后进入正常状态页。

### 6.2 第 1 行状态说明

| 第 1 行内容 | 含义 |
| ---- | ---- |
| `STATE: LOCKED` | 当前为锁定状态，可以执行开锁 |
| `STATE: RIDING` | 当前为解锁状态，表示正在使用中，可以执行关锁 |
| `STATE: BUSY` | 正在执行开锁或关锁动作，短时间内不接受新的开锁/关锁/播放指令 |

### 6.3 第 2 行动作说明

| 第 2 行内容 | 触发场景 | 含义 |
| ---- | ---- | ---- |
| `LAST: READY` | 上电初始化结束 | 固件已进入待命状态 |
| `LAST: S 0` | 收到状态查询且当前锁定 | 返回 `S,0` |
| `LAST: S 1` | 收到状态查询且当前解锁 | 返回 `S,1` |
| `LAST: U RUN` | 正在执行开锁 | 舵机和提示音正在开锁流程中 |
| `LAST: U OK` | 开锁完成 | 返回 `U,OK` |
| `LAST: U ALREADY` | 已经处于解锁状态时再次收到开锁 | 返回 `U,ALREADY`，并播放提示音 `002` |
| `LAST: U BUSY` | 忙状态下收到开锁 | 返回 `U,BUSY` |
| `LAST: L RUN` | 正在执行关锁 | 舵机和提示音正在关锁流程中 |
| `LAST: L OK` | 关锁完成 | 返回 `L,OK` |
| `LAST: L ALREADY` | 已经处于锁定状态时再次收到关锁 | 返回 `L,ALREADY`，并播放提示音 `004` |
| `LAST: L BUSY` | 忙状态下收到关锁 | 返回 `L,BUSY` |
| `LAST: P1` | 收到 `P,1` | 播放曲目 `001` |
| `LAST: P2` | 收到 `P,2` | 播放曲目 `002` |
| `LAST: P3` | 收到 `P,3` | 播放曲目 `003` |
| `LAST: P4` | 收到 `P,4` | 播放曲目 `004` |
| `LAST: P5` | 收到 `P,5` | 播放曲目 `005` |
| `LAST: P BAD` | 收到 `P,0`、`P,6`、`P,9` 或其他非法 `P,n` | 返回 `P,BAD` |
| `LAST: P BUSY` | 忙状态下收到 `P,n` | 返回 `P,BUSY` |
| `LAST: BAD CMD` | 收到不支持的文本 | 返回 `E,BAD` |

### 6.4 LCD 使用注意事项

- LCD 不显示中文。
- LCD 第 2 行只保留最近一次事件，不保存历史记录。
- 当第 1 行显示 `STATE: BUSY` 时，第 2 行通常会显示 `LAST: U RUN` 或 `LAST: L RUN`。

## 7. 提示音曲目映射

| 曲目编号 | 文件建议名 | 当前用途 |
| ---- | ---- | ---- |
| `001` | `001.xxx` | 开锁成功提示音 |
| `002` | `002.xxx` | 开锁重复或开锁失败提示音 |
| `003` | `003.xxx` | 关锁成功提示音 |
| `004` | `004.xxx` | 关锁重复或关锁失败提示音 |
| `005` | `005.xxx` | 围栏拒绝提示音 |

说明：

- 当前代码默认通过 `P,1` 到 `P,5` 对应播放以上 5 个曲目。
- 文件放在 TF 卡根目录。
- 曲目编号仍受模块实际文件排序规则影响，建议测试卡中只保留需要的音频文件。

## 8. 蓝牙协议对应关系

蓝牙协议详细说明见：

- `docs/bluetooth_protocol.md`

当前整机协议的核心要点是：

- `U`：开锁
- `L`：关锁
- `S`：查询状态
- `P,n`：播放指定提示音
- 支持 `\n`、`\r`、`\r\n`
- `U/L/S` 和固定长度 `P,数字` 也支持无换行直接识别

## 9. 调试建议

### 9.1 单片机联调

推荐顺序：

1. 先用 App 连接蓝牙
2. 查询状态，确认 `S,0` 或 `S,1`
3. 测试 `P,1`
4. 测试开锁
5. 测试关锁

### 9.2 App 联调

- 若手机定位不可用：
  - 可在调试页设置模拟当前位置
  - 可临时关闭围栏校验
- 若需要测试提示音：
  - 可直接使用调试页的 `播放 001-005`
- 若协议有异常：
  - 可查看错误面板并复制详情

## 10. 当前实现边界

- GPS 在整机流程中不参与任何逻辑。
- 单片机不保存骑行记录。
- 围栏判断完全在 App 侧完成。
- 单片机仅维护当前锁状态，不理解历史骑行语义。
- LCD 仅用于本地状态展示，不与 App 做复杂同步。
